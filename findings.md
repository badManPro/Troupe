# Findings

- `requirements` 阶段当前只有角色提示词，没有前端层的“阶段任务说明”。用户进入后看到的仍是通用欢迎语，因此不知道该聊什么。
- 从现有角色定义看，需求定义阶段本质上分成两步：PM 负责把头脑风暴收束成 PRD 与 MVP 边界，QA 负责补边界场景、验收标准和风险点。
- 这个问题不需要改数据库或新建阶段实体。只要在聊天空态里加入阶段化指引，并提供可直接发送的起手 prompt，就能显著降低用户启动成本。
- 仅靠系统提示词不够，因为用户在第一眼接触界面时还没开始对话。需求定义的“模块目标 / 讨论内容 / 阶段产出”必须在 UI 中直接可见。
- 输入框 placeholder 也需要阶段化，否则用户仍然会面对“输入你的想法或回复...”这种过于泛化的提示。
